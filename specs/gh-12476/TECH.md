## Proposed changes
All changes are in the `warp` app crate; no server or API changes (PRODUCT Non-goals).

### Filter model (`app/src/ai/agent_conversations_model.rs`)
- New types next to `EnvironmentFilter`:
  - `RepositoryIdentity`: `OwnerRepo(GithubRepo)` | `GitRoot(String)` (standardized absolute root path). Implement a dedicated `matches(&self, other) -> bool` helper: case-insensitive on owner+repo for `OwnerRepo` (PRODUCT 9, 12), exact path equality for `GitRoot` (PRODUCT 11). Do not change `GithubRepo`'s derived case-sensitive `PartialEq` — handoff env-overlap matching depends on it.
  - `RepositoryFilter`: `#[default] All` | `NoRepository` | `Specific(RepositoryIdentity)`, derived Serde like its siblings.
- Add `#[serde(default)] pub repository: RepositoryFilter` to `AgentManagementFilters`; extend `reset_all_but_owner` and `is_filtering` (PRODUCT 2, 34; old persisted blobs deserialize to `All`, PRODUCT 32).
- Add `FilterType::Repository` to [`app/src/ai/agent_management/telemetry.rs:46`](https://github.com/warpdotdev/Warp/blob/ae832ff68503fd3f97556f5193dc359d0478d007/app/src/ai/agent_management/telemetry.rs#L44-L53).

### Directory→repository resolver (new `app/src/ai/agent_management/repo_resolver.rs`)
The one new abstraction. A `SingletonEntity` model owning `HashMap<String, RepoResolution>` where `RepoResolution = Pending | Unresolvable | Resolved(Option<RepositoryIdentity>)`:
- `resolved(&self, dir) -> Option<&Option<RepositoryIdentity>>` — synchronous cache read used during matching; `None` means still unresolved (PRODUCT 27).
- `request(&mut self, dir, ctx)` — inserts `Pending` and spawns `resolve_repo_for_path` (reused from `touched_repos.rs`; same crate, widen to `pub(crate)` export via `handoff/mod.rs` if needed). On completion, store `Resolved` and `ctx.emit` a change event; `AgentManagementView` subscribes and re-runs `get_tasks_from_model` so rows appear/disappear without selection changes (PRODUCT 27, 30).
- `RepoResolution` includes a terminal `Unresolvable` state, and a source/session check runs before a recorded path ever touches the current filesystem (PRODUCT 19). A directory is eligible for local resolution only when (a) it was recorded by this Warp installation — its backing conversation metadata has `has_local_data`, whereas [`AIConversationMetadata::from_server_metadata` at `app/src/ai/blocklist/history_model.rs:130`](https://github.com/warpdotdev/Warp/blob/ae832ff68503fd3f97556f5193dc359d0478d007/app/src/ai/blocklist/history_model.rs#L130-L166) carries a server-recorded `working_directory` that may originate from another machine and is never resolved locally — and (b) the entry has no `environment_id`, since cloud-run directories describe the environment's filesystem. Ineligible directories, non-absolute paths, and absolute paths that do not exist as a local directory (e.g. an unavailable remote SSH session's directory) store `Unresolvable`: such entries match neither a concrete repository nor No repository (PRODUCT 19, 27). Only a completed local resolution may produce `Resolved(None)` and No repository membership (PRODUCT 26). Failed/timed-out git calls resolve to `GitRoot` identity when a root was found, else `Resolved(None)` — never silently broaden (PRODUCT 31).
- The resolver and local association compile only under the `local_fs` feature — the handoff module that exports `resolve_repo_for_path` is already `#[cfg(feature = "local_fs")]`-gated. On wasm builds the dropdown still renders with environment-metadata association only, and Current repository stays in its unavailable state.

### Entry association and matching (`entry.rs`)
- Add `initial_working_directory: Option<String>` to `AgentConversationDisplayData` so both directories from PRODUCT 13 are available (task entries already use initial at `entry.rs:517`; conversation entries add `nav_data.initial_working_directory` alongside the existing latest-fallback field at `entry.rs:627`).
- Add `matches_repository(&self, filter, app)` to the `matches_filters` chain:
  - `All` → true.
  - Local association: resolver lookups for the entry's initial + latest directories (deduped); directories are submitted only when the entry passes the resolver's source/session gate (locally recorded metadata, no `environment_id`).
  - Cloud association: the union of (a) `display.environment_id` → `CloudAmbientAgentEnvironment::get_by_id` → `github_repos`, mirroring `matches_environment` + the env-name lookup (PRODUCT 21 — inaccessible env objects yield no association from this path), and (b) already-available run metadata: `owner/repo` identities parsed from the run's PR artifact URLs ([`Artifact::PullRequest` at `app/src/ai/artifacts/mod.rs:38`](https://github.com/warpdotdev/Warp/blob/ae832ff68503fd3f97556f5193dc359d0478d007/app/src/ai/artifacts/mod.rs#L37-L45), already on `display.artifacts`; the existing [`parse_github_pr_url` at `mod.rs:253`](https://github.com/warpdotdev/Warp/blob/ae832ff68503fd3f97556f5193dc359d0478d007/app/src/ai/artifacts/mod.rs#L251-L264) keeps only the repo name, so add a sibling URL→`GithubRepo { owner, repo }` helper). Run-metadata association applies to run-backed entries even when no environment object resolves, so listed cloud runs with a deleted, inaccessible, or absent environment still match (PRODUCT 16); PR URLs are structured run metadata, not title/branch guessing (PRODUCT 20).
  - `Specific(id)` → any association matches (PRODUCT 17, 18); `NoRepository` → all lookups resolved and empty (PRODUCT 26); unresolved → false for both (PRODUCT 27).
- Matching never triggers I/O. The submission data path: `AgentConversationsModel` gains a small accessor returning deduped initial/latest directories for currently listable entries that pass the source/session gate (same shape as `get_all_environment_ids_and_names`, `agent_conversations_model.rs:1717`); `AgentManagementView::get_tasks_from_model` feeds them to the resolver's `request_missing` before calling `get_entries`, and the resolver's change event re-runs `get_tasks_from_model` (PRODUCT 27, 30).

## Parallelization
Not proposed for implementation. The change is one dependency chain inside the `warp` crate — filter types → resolver → entry matching → view wiring — with the model and view edits touching the same files other filters touch (`agent_conversations_model.rs`, `entry.rs`, `view.rs`). Splitting it across agents would serialize on type definitions and merge conflicts rather than save wall-clock time. A single optional local validation agent (own worktree, e.g. `git worktree add ../warp-gh12476-validate`) can run `cargo nextest` + presubmit while manual validation proceeds, if desired.

## Risks and mitigations
- Wasm builds: local git resolution is `local_fs`-gated; the env-metadata-only fallback above keeps the view compiling and useful on web. Verify with the workspace build in presubmit.
- The filter evaluates only client-loaded entries; server task fetches are unaffected (Harness-filter precedent), so unloaded older cloud runs appear once fetched rather than via a server-side repo query.
- Git invocations per unique directory: bounded by the resolver cache (one resolution per directory per session) and the existing 5s timeout in `touched_repos.rs`; resolution is async and never blocks render (PRODUCT 30).
- Recorded SSH-pane directories carry no session marker today ([`TODO(CORE-3546)` at `app/src/ai/agent/mod.rs:2990`](https://github.com/warpdotdev/Warp/blob/ae832ff68503fd3f97556f5193dc359d0478d007/app/src/ai/agent/mod.rs#L2990)), so a locally persisted conversation's remote path that coincides with an existing local Git repo can still mis-associate. Accepted for v1: the local-existence and git-root requirements bound the exposure; recording shell/session provenance at capture time is the durable fix.
- Case-insensitive matching diverging from handoff's case-sensitive env overlap: isolated in `RepositoryIdentity::matches`; `GithubRepo::PartialEq` untouched.
- Stale associations after a repo's remote changes: acceptable for v1 — cache lives for the app session only.
- Rollout: feature flag default-on for dogfood only; removing the flag later follows the standard promote/remove skill flow.
